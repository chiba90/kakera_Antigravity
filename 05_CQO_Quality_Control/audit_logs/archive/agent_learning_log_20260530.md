# AI Agent Learning & Update Log

このファイルは、ユーザーからの指摘事項（フィードバック）や、システム・ワークフローへの変更履歴を記録し、同じミスを繰り返さないための自律的学習ログです。
AIエージェントのタスク着手前やコード修正前にこのログを参照し、過去の失敗と改善措置を必ず確認してください。

## 📥 読み込み（Read）と 📤 記録（Write）のタイミング
AIは以下のタイミングを絶対のトリガーとして、本ファイルを運用すること。

**【📥 読み込み（参照）のタイミング：作業の「前」】**
1. ユーザーから新たなタスク（執筆、リライト、スクリプト作成など）を依頼された直後。
2. 既存のルールやワークフロー、アーキテクチャ定義を変更・修正しようとする前。

**【📤 書き込み（記録）のタイミング：事象の「直後」】**
1. ユーザーから明確な「ダメ出し」や「修正指示（ここはこうして等）」を受け、その修正対応が完了した直後。
2. AI自身の提案や出力がエラー・仕様不適合を起こし、その根本原因と解決策が判明した直後。
3. ルールファイルやワークフロー（`.agent/rules/` や `.agent/workflows/` 等）の構造を変更・アップデートしたセッションの終端。
- ※ 追記時は、必ず**降順（最新の記録が一番上）**になるように記載すること。

## 📝 【2026-05-30】「国産い草」セッションでの先走り（進行プロセス違反）とガバナンス改善策

*   **事象:** 社長から「記事に嘘はないか」「内部リンクのリンク先は本当に存在しているか」という厳格なQA監査のご指示をいただいたにもかかわらず、その確認報告を提示して承認を得る前に、COOが先走って次の画像生成ステップ（Step 4）を開始してしまった。社長より「記事の完成度のチェックを頼んだのに、なぜ画像作成に進んだのか。COOとして問題」と厳しいお叱りを受けた。
*   **根本原因:** ワークフローの「自動起動」という手続き的な進行に盲従し、社長が最も懸念されている「コンテンツの誠実さ（事実と品質への保証）」をご報告してご安心いただく、という人間中心のガバナンス（合意形成）のプロセスを完全に無視してしまったため。
*   **【恒久策】:** 今後、社長から「ファクトチェック」や「品質・完成度の検証」を求められた際は、自律的に一切の次工程（画像生成、コーディング、結合など）の動作を完全に凍結（フリーズ）させること。実データ（CSVとの照合、科学的ファクトのソース明示）を100%揃えた検証報告書をチャットに提示し、社長から「検証結果を確認、次の工程へ進んでよし」という明示的な決裁を獲得するまでは、いかなる場合でも絶対に次の物理的なツールの呼び出し（generate_image等）を行ってはならない。

## 📝 【2026-05-28】「ロックマン江戸切子」セッションでの致命的品質バグと恒久的改善策

### 1. 現物アセット（画像）の検証不足（トリミング漏れ）
*   **事象:** 前回のクロップ処理が別パスで実行されていたため、納品コピー時にトリミング前の「生画像アセット」をコピーして上書き納品してしまい、社長から「トリミングが全然できていない」と厳しく指摘された。
*   **根本原因:** スクリプトの実行成功ログだけを過信し、納品先フォルダ内の物理画像の実際の寸法（Width/Height）を自律的にツール測定して確認する裏取り（QA）を完全に怠ったため。
*   **【恒久策】:** 今後、画像アセットをトリミング・配置した際は、必ず直後に **PowerShell等を用いて納品フォルダ内の実画像サイズを物理測定し、16:9比率（1024x576など）を満たしていることを実データで確認する検証プロセスを強制義務化**する。

### 2. チャット出力バグによる古いノイズ（of）の再混入
*   **事象:** ローカルファイル上では `of` を完全に「の」に修正していたにもかかわらず、社長にコピペ用として提示したチャット上のHTMLコード内に、修正前のドラフトテキストが誤って再混入（ゾンビ化）した状態で出力され、表示崩れや `of` の残存を引き起こした。
*   **根本原因:** コンテキスト内に新旧の文章が混在し、1万文字もの長大なコードをチャットに直接出力しようとしたため、モデルの出力デコード時にハルシネーションが発生した。
*   **【恒久策】:** 1万文字を超えるような巨大なコード全体をチャット欄に直貼りすることを完全に禁止する。今後は**「ローカルの物理ファイル（SSoT）を直接完璧に修正・保存し、そのファイルリンクからコピーしてもらう」**運用に統一する。また、提示前に grep や正規表現で `of` や不自然なタイポを自律スキャンする検閲ゲートを設置する。

### 3. 表示崩れ（縦書き演出のテーマ干渉）
*   **事象:** 詩的演出のために使用した `writing-mode: vertical-rl` のインラインスタイルが、WordPressテーマの親要素の幅制限と衝突し、コンテンツ全体を画面外に押し出して非表示（ホワイトアウト）にするバグを引き起こした。
*   **根本原因:** テーマやブラウザ環境との互換性を検証せず、極端なインラインスタイルによるトリッキーな装飾を施したため。
*   **【恒久策】:** `writing-mode` などのテーマ干渉リスクが極めて高いインラインCSSの使用を原則禁止し、どんなWordPress環境でも絶対に崩れない「和風モダンな美しい境界線つきセンタリング（横書き）ブロック」などの安全で高互換なスタイルに置き換える。

### 4. 係り受け・タイポの検知漏れ
*   **事象:** 守破離の解説ブロックにおいて `キャラクターの宿宿` という、日本語として完全に破綻したタイポの検知を漏らしたまま納品した。
*   **根本原因:** LLM生成時のバグを、自律的な日本語の目視精読チェックを行わずにそのままリリースしたため。
*   **【恒久策】:** 納品前に、各見出しやリストブロック、テーブル内のテキストについて、日本語としての係り受けや重複ワード（「宿宿」等）がないかを、人間（編集者）の目線で徹底的に精読・セルフ校正するステップを強制する。
